# Manual end-to-end testing

The backend's automated tests (see [Conventions](./conventions.mdx#testing)) cover individual
pipeline stages with the model runtime mocked or stubbed. Checking the whole pipeline, ingestion
through extraction, project resolution, retrieval, and chat, against a real Ollama instance
needs a small corpus of realistic documents. This page describes how to build that corpus and
what to check once it is ingested.

## Where the corpus lives

`test-data/` at the repo root is gitignored and exists for exactly this: local documents for
manual ingestion testing (see the root `.gitignore`). There is no fixed corpus checked into the
repository, build one that exercises the pipeline stage you are verifying.

Re-running a corpus from a clean state (for example after changing the ingestion order, or
re-checking a resolution outcome) no longer needs a manual `TRUNCATE`: the sidebar's [Clear
database](../reference/frontend.mdx#clear-database) button wipes every table via [`POST
/admin/reset`](../reference/backend.mdx#post-adminreset).

## Categories of test documents

A corpus that exercises the full pipeline, including all three [project resolution
outcomes](../architecture/memory-model.mdx#incremental-updates-and-entity-resolution), benefits
from documents in these categories:

1. **Base project**: one or two documents describing a single project (for example a PRD and a
   technical concept document) with named people, teams, and terms. The first document ingested
   for a project should resolve as `"new"`.
2. **Follow-up document for the same project**: a status update or retro that describes the same
   project under a similar name and description, and reuses the same team and people names.
   Should resolve as `"match"`, and should reuse the existing `Person`/`Team` rows rather than
   creating duplicates.
3. **A second, unrelated project**: different name, teams, people, and terms. Its first document
   should also resolve as `"new"`; a follow-up document for it exercises `"match"` for a second
   project, and lets you check that retrieval and chat keep the two projects' contexts separate.
4. **An ambiguous document**: a document whose extracted project name and description plausibly
   overlap two existing projects (it can reuse concepts and teams from both). Exercises the
   `"ambiguous"` path: a pending `ProjectResolution` row, `GET /project-resolutions`, and both
   branches of `POST /project-resolutions/{id}/resolve` (attach to an existing project, or create
   a new one).
5. **A PDF document**: similar content to the markdown documents, to exercise the PDF parser's
   page-based chunking.
6. **An `.mdx` document**: verifies `.mdx` is ingested the same as `.md`.
7. **A versioned pair of the same document (v1 then v2)**: ingest the same file path twice,
   unchanged the first time (skipped via the content hash) and then with changes (people, terms,
   or relations added or changed), to verify the document's `Term`/`Team`/`Person`/`Relation`
   rows are replaced, not duplicated, on re-ingestion.
8. **A cross-cutting or glossary document**: not centered on a single project, with relation
   triples that reference concepts which are not people, teams, or projects, exercising
   `resolve_entity_ref()`'s fallback to creating a `Topic`.

## Resolution outcomes are LLM-dependent

`resolve_project()` and extraction both go through the configured LLM (see [model
providers](../architecture/model-providers.mdx)), so the outcome for categories 3 and 4 above
(`"new"` vs `"match"` vs `"ambiguous"`) is not guaranteed by the input alone. Treat the expected
outcome as a hypothesis to check against `GET /projects` and `GET /project-resolutions`, and
record what actually happened rather than assuming the design intent held.

## Ingestion order matters

Project resolution compares the extracted candidate against `Project` rows that already exist at
ingestion time. Ingest the base and unrelated-project documents (categories 1 and 3) before any
document meant to test `"match"` or `"ambiguous"` against them.

## Chat prompts to check retrieval and grounding

Once the corpus is ingested, send prompts to `POST /chat` (or the Electron frontend) that
exercise different retrieval scenarios:

- A question whose answer sits in a single chunk of one document, then check the answer is
  grounded and the cited source document and section match.
- A question whose answer requires combining several chunks from the same project's documents.
- A question about a term or entity introduced only in the PDF (category 5) or `.mdx` (category
  6) document, to confirm both formats are retrievable.
- A question that spans two projects, since `search_chunks()` is not scoped to a project, then
  check the answer combines information from both correctly and cites both sources.
- After re-ingesting the v2 of category 7's document, repeat a question whose answer changed (for
  example "who is on team X"), and check the answer reflects the new content, not the old.
- A question with no answer in the corpus, then check the model says so rather than fabricating
  an answer, per the grounding instructions in `app/chat/prompts.py`.

## Testing chat attachments

[Chat attachments](../architecture/memory-model.mdx#chat-attachments) are a separate path from
`POST /documents/ingest` and worth checking on their own, ideally with a couple of small,
throwaway files distinct from the main corpus so promoting them does not interfere with the
resolution outcomes tracked for the categories above:

1. Attach a single file to a chat message and ask a question about its content. Check the answer
   cites it (`is_attachment: true` in `sources`), and that `GET /projects` does **not** show a new
   project yet, no extraction has run.
2. Ask a second, unrelated question in the same conversation without re-attaching. Check the
   answer can still use the attachment (it is scoped to the conversation, not just the message it
   was attached to).
3. Start a **new** conversation and ask about the same file's content without attaching it. Check
   the answer does not use it, an unpromoted attachment must not leak into another conversation or
   into corpus-wide retrieval.
4. Click "save to memory" on the attachment. Check it runs [project
   resolution](../architecture/memory-model.mdx#incremental-updates-and-entity-resolution) the
   same as a directly-ingested document (including landing in `GET /project-resolutions` if
   ambiguous), and that a fresh conversation can now retrieve it without attaching it again.
5. Attach a folder, or several files, to one message and check every supported file inside it
   becomes a separate attachment. After the answer, use "save all to memory" and check each one
   resolves independently, later ones in the batch can land `"match"` against a project an
   earlier one in the same batch just created.

## Testing conversation history

[Conversation history](../reference/frontend.mdx#conversation-history) is sidebar UI state built
on the `Conversation`/`Message` rows every chat turn already writes, so it is mostly a matter of
checking the sidebar reflects what the backend has:

1. Send a message in a fresh chat (`#sidebar-history` starts on "No history yet"). Check a titled
   entry appears in the sidebar once the answer comes back, without reloading the app.
2. Click "New Chat", send a message in a second conversation. Check a second entry appears above
   the first (most recently active first) and the composer no longer carries the first
   conversation's `conversationId`.
3. Click the first (older) entry. Check its messages replay in `#chat` and it becomes the
   highlighted entry. Send a follow-up. Check no third entry appears, the reply must extend the
   resumed conversation, not start a new one; the resumed entry's preview does not itself update
   until the app is restarted (see [what's not here yet](../reference/frontend.mdx#what-is-not-here-yet)).
4. Attach a file to a message, then resume that conversation later. Check the attachment's
   filename is listed when the conversation reloads.
5. Open Settings (sidebar) → Database, click "Clear database". Check the sidebar returns to
   "No history yet" alongside the rest of the reset.

## Testing the Settings modal

The [Settings modal](../reference/frontend.mdx#settings-modal) is UI-only outside of Database, so
there is nothing to verify against the backend beyond Clear database (covered above). Worth a
quick visual pass after touching it: open each of the five sections and check the active-state
highlight follows the left-nav selection; open a model dropdown (Models section) and check the
same choices appear whether opened from the default selector or any of the four per-task
overrides; open the YouTrack or Slack section and click the eye icon on a secret field to confirm
the show/hide toggle works.

## Tracking a concrete corpus

A worked example following this structure can keep its own checklist as `test-data/TEST-PLAN.md`
(also gitignored): file names, the expected `project_resolution` for each, and the concrete list
of chat prompts to run against that corpus. Keep it next to the corpus it describes and update it
as the corpus changes, it is the checklist for a specific local run, not a fixed part of the
pipeline.
