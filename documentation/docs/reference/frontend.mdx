# Frontend

The frontend is the Electron app in `frontend/src/`. It has no bundler: `npm start` first
rebuilds `src/tailwind.css` (`npm run build:css`) and then runs `electron .`, which loads the
files in `src/` directly.

## Tech stack

The renderer is a plain web page with no framework:

- **A hand-written design system in `style.css`**, ported from the approved phase-1 design demo.
  All design tokens live in its `:root` block: the palette (background tiers, hairline colors,
  four text tiers, a teal accent, amber for decisions/memory, red for destructive/errors), a
  spacing scale of exactly 4/8/12/16/24/32/48px, and exactly three radii (2px chips/tags, 4px
  fields/cards, 6px overlay panels).
- **Tailwind CSS** (built locally into `tailwind.css`) contributes only its preflight reset;
  `tailwind.config.js` defines no theme tokens, so there is a single source of design tokens.
- **Inline SVG icons** (sidebar toggle, settings gear, file, plus, send, chevron), copied from
  the design demo. Nothing loads from the network.
- **Geist** (400/500/600) for prose and UI labels; **JetBrains Mono** (400/500) for everything
  meta/structural: gutter labels, tags, timestamps, source manifests, and action-button copy.
  Both are self-hosted in `src/fonts/`.

Copy conventions carried over from the design review: action buttons are lowercase mono ("save
to memory", "new session"), section labels are sentence case ("Sources", "Sessions"), and no
dots or middle dots are used as text separators (structure comes from gaps and color tiers).

## Processes

- **Main process** (`main.js`): on `app.whenReady()`, creates a single 1280x820 `BrowserWindow`
  (menu bar hidden, `backgroundColor: "#0e0e0e"` to match the dark theme and avoid edge artifacts
  on resize) with `preload: preload.js`, `contextIsolation: true`, `nodeIntegration: false`, loads
  `index.html`, and sets the zoom factor to 1.15 once it finishes loading. Quits when all windows
  are closed, except on macOS (`process.platform !==
  "darwin"`). Also registers two `ipcMain.handle` handlers:
  - `documents:select-paths`: opens a native dialog (`dialog.showOpenDialog` with `properties:
    ["openFile", "openDirectory", "multiSelections"]`) and returns the chosen paths, or `[]` if
    the dialog is canceled.
  - `documents:list-files`: given a file or folder path, returns the flat list of supported files
    it resolves to (`listSupportedFiles()`, a recursive walk filtered to `.md`/`.mdx`/`.pdf`,
    mirroring the backend's `discover_files()`/`SUPPORTED_EXTENSIONS`) or `[]` if the path is
    invalid; this is how a dropped or picked folder becomes one chip per file, see
    [Attachments](#attachments-rendererjs).
- **Preload script** (`preload.js`): calls `contextBridge.exposeInMainWorld("openproject", {
  ... })`, exposing to the renderer:
  - `backendUrl`: the `OPENPROJECT_BACKEND_URL` environment variable, or `"http://localhost:8000"`
    if unset.
  - `selectPaths()`: invokes `documents:select-paths` IPC and resolves to the array of chosen
    paths (or `[]` on cancel).
  - `listFiles(path)`: invokes `documents:list-files` IPC for one path.
  - `getPathForFile(file)`: wraps `webUtils.getPathForFile()` to resolve a dropped `File` object
    to its absolute filesystem path.
- **Renderer** (`index.html`, `renderer.js`, `style.css`): the chat UI, running as a regular web
  page with no Node/Electron access beyond what `preload.js` exposes.

## UI structure

`index.html` composes the window from three regions plus overlays:

- **Top status bar** (`.topbar`, 44px): a sidebar toggle button (`#toggle-sidebar-btn`), the
  `openproject` wordmark (mono, with a teal diamond), the active session title centered
  (`#topbar-name`, lowercase, "new session" when none), and a settings gear (`#settings-btn`).
- **Sessions sidebar** (`.sidebar`, 232px, collapsible): a "new session" button
  (`#new-chat-btn`), a "Sessions" label, and the `#sidebar-history` list, see
  [Conversation history](#conversation-history). Collapsing toggles `sidebar-hidden` on
  `<body>`, via the topbar button or Ctrl/Cmd+B. There is no help/feedback footer.
- **Conversation column**: `main#chat` is the scroll area; inside it a centered 720px `.column`
  holds the `#messages` stream. The stream is one ledger: every entry is a two-column grid
  (`.entry`, `[88px gutter | content]`) split by a single continuous vertical hairline
  (`border-left` on `.entry-body`). The gutter names the actor of every entry: `You` (accent,
  with the send time under it), `Assistant`, or an event tag (`Memory`, `Fact`, `Resolve`,
  `System`, `Error`). User and assistant text share identical typography; a user entry is
  marked only by its gutter label and by its segment of the vertical rule turning
  accent-colored.
- **Composer** (`.composer-zone` under the scroll area, inside the same 720px column): a
  bordered field (`#composer`) with a staged-attachments row (`#pending-attachments`, hidden
  via CSS `:empty` when nothing is staged), a plus icon button (`#attach`), the auto-growing
  `#message` textarea, and an icon send button. Focus turns the border teal; a drag-over turns
  it accent (`body.dragover`).

Overlays (all `.overlay` divs, hidden by default): the history search palette
(`#history-overlay`), the settings modal (`#settings-modal`), and the clear-database
confirmation (`#clear-database-modal`). Clicking an overlay's backdrop closes it, as does
Escape (with an in-flight guard for clear-database, see [Clear database](#clear-database)).

## Keyboard shortcuts

- **Enter / Shift+Enter** in the composer: send / newline.
- **Ctrl/Cmd+B**: toggle the sessions sidebar.
- **Ctrl/Cmd+K**: open the history search palette (see
  [Conversation history](#conversation-history)); it is keyboard-only, there is no visible
  search button.
- **Escape**: close any open overlay.
- In the palette: **ArrowUp/ArrowDown** move the active row, **Enter** opens it.

## Startup

On DOM ready, the renderer runs three calls in parallel:

- `loadPendingResolutions()`: fetches and renders any `ProjectResolution` rows left pending from
  a previous session (see [Ambiguous project resolution](#ambiguous-project-resolution-rendererjs)).
- `loadPendingFacts()`: fetches and renders any pending fact confirmations (see
  [Fact confirmation](#fact-confirmation-rendererjs)).
- `loadConversationHistory()`: fetches and renders the sidebar's session list (see
  [Conversation history](#conversation-history)).

## New session

Clicking `#new-chat-btn` (the sidebar's "new session" button) runs `resetConversationView()`:
clears `messages.innerHTML`, resets `conversationId` to `null` and `pendingAttachments` to `[]`,
clears the active highlight from the session list, and resets the topbar title to "new session".
The backend keeps the old conversation's rows, but the renderer no longer sends their
`conversation_id`, so replies start a fresh thread; the app never auto-resumes the last
conversation on launch, only on an explicit click in the history list or palette.

## Conversation history

`loadConversationHistory()` fetches [`GET /conversations`](./backend.mdx#get-conversations) (via
the same `fetchAllPages()` helper used for pending resolutions/facts/projects) into a
module-level `conversations` array, shared by the sidebar and the palette, and renders the
sidebar via `renderHistoryList()`: one `.sess` button per conversation, in the order the backend
returns them (most recently active first). A `.sess` shows the conversation's `title` (falling
back to `"New conversation"`), a short date from `updated_at` (`formatSessionDate()`: "today",
"yesterday", then "jul 06"-style lowercase month + day), and the mono `preview`, all truncated
with CSS. The active session gets an inset accent bar (`.sess.sel`). An empty list renders a
"no sessions yet" placeholder.

Clicking a `.sess` calls `openConversation(id)`: fetches [`GET
/conversations/{id}`](./backend.mdx#get-conversationsid), clears `messages.innerHTML`, sets
`conversationId = data.id` and `pendingAttachments = []`, renders one `System` entry listing any
attachment filenames if `data.attachments` is non-empty, then replays `data.messages` in order
(assistant messages via `addAssistantEntry()` so their citations and sources render exactly like
live answers; replayed user entries show no time, since the API has no per-message timestamps),
marks the session active, and sets the topbar title. From this point, `conversationId` is set,
so the next composer submit continues this conversation rather than starting a new one.

The **history search palette** (Ctrl/Cmd+K) is a client-side filter over the same
`conversations` array: `renderPalette(filter)` matches the query against titles and previews and
renders `.palette-item` rows (title, date, preview). ArrowUp/ArrowDown move the `.active` row,
Enter or a click opens it via the same `openConversation(id)` and closes the palette. Opening
the palette clears the previous query.

After a successful `POST /chat` response for a brand-new conversation (`conversationId` was
`null` when the request was sent) that came back with a non-null `title`, the chat flow calls
`prependHistoryItem()` to unshift the new conversation into `conversations`, re-render the
sidebar with it active, and set the topbar title. This is the only case the list updates live:
continuing an existing conversation does not move or refresh its entry (its
`preview`/`updated_at` go stale until the app is next restarted or the list is otherwise
reloaded).

## Settings modal

The topbar's gear button opens `#settings-modal`, a 720x520 two-pane panel (`.settings-panel`)
styled like the rest of the app: a left-hand section list (`.snav` buttons), a right-hand
content pane (`.spanel` divs). `setActiveSettingsSection(section)` toggles the `sel` class on
the matching nav button and panel; the modal always opens back on the Models section. It closes
via the backdrop or Escape (there is no close button). Five sections: **Models**, **YouTrack**,
**Slack**, **Statistics**, **Database**.

Models, Statistics, and Database are wired to real backend calls; YouTrack and Slack remain UI
scaffolding with no persistence, static placeholder forms standing in for a future settings API:

- **Models**: a default-model dropdown plus per-task overrides (Chat, Extraction, Embeddings,
  Orchestration). A native `<select>`'s open list can't be restyled to match the app's dark
  theme, so these are a custom combobox (`.model-dropdown`): a trigger button plus an
  absolutely-positioned `.model-dropdown-menu`. `initModelDropdowns()` runs once at script load
  (not on every modal open, since the elements exist in the DOM regardless of the modal's
  visibility and re-running it would double-attach listeners): it fetches [`GET
  /settings/models`](./backend.mdx#get-settingsmodels) and, for each `.model-dropdown`, either
  renders it editable (`initEditableModelDropdown()`, options from the response's
  `available_llm_models`, a leading "use default" option for the per-task ones, none for
  Default model) or, for the one `data-task="embeddings"` container, calls
  `renderEmbeddingsRow()` to show `embeddings_model` as plain disabled text with no menu (see
  [Selection and configuration](./llm-integration.mdx#selection-and-configuration) for why the
  embedding model isn't editable from here). If the initial fetch fails, every trigger is
  disabled, its value set to "unavailable", and the error is shown in `#model-settings-error`.
  Selecting an option autosaves: `setModelDropdownValue()` updates the UI optimistically,
  `#model-settings-status` shows "saving…" then "saved" (auto-clearing after 2s), and a `PATCH
  /settings/models` failure reverts the dropdown to its previous value and shows the error in
  `#model-settings-error` instead. `closeAllModelDropdowns()` runs on any document click, so
  opening one closes the others and clicking outside closes all of them. There is no cloud
  provider option group: none is implemented yet (see [model
  providers](../architecture/model-providers.mdx#hosted-providers)).
- **YouTrack** / **Slack**: static forms (URL/token/default-project; workspace/bot-token/signing-
  secret/default-channel) with a "not connected" status pill and password-style secret inputs
  with a text show/hide toggle (`[data-toggle-password]`, a mono "show"/"hide" button inside the
  field). "test connection" and "save" render but do nothing.
- **Statistics**: plain label/value rows (`.stat-row`, `[data-stat]` spans), grouped under
  "Corpus" (Projects, Documents, Chunks, Conversations) and "Usage" (Model calls, Tokens used)
  labels. `loadStats()` runs once at script load (like `initModelDropdowns()`, regardless of
  whether the modal is ever opened): it fetches [`GET /stats`](./backend.mdx#get-stats) and
  fills in each `[data-stat]` element via `formatStatCount()` (plain integer under 1,000,
  otherwise `"3.5K"` / `"2.1M"`-style abbreviation). A failed fetch shows the error in
  `#stats-error` and leaves every value as its placeholder `"–"`.
- **Database**: see [Clear database](#clear-database) below.

## Clear database

The Settings modal's Database section holds a danger zone with a "clear database" button that
opens the confirmation overlay (`#clear-database-modal`) warning that the action permanently
deletes all projects, documents, and conversation history. Once confirm is clicked, Cancel, the
backdrop, and Escape are all inert (a module-level `clearDatabaseInFlight` flag) until the
request settles, since the backend call is already dispatched by then and cannot actually be
cancelled. On success ([`POST /admin/reset`](./backend.mdx#post-adminreset)), it closes both the
confirmation and the settings modal, runs the same `resetConversationView()` used by
[New session](#new-session), empties the `conversations` array (so the sidebar and palette show
their empty states), and appends a `System` entry, "database cleared, all local state reset".
On failure, the error is shown inline in the modal (`#clear-database-error`) and the modal stays
open so the user can retry.

## Chat flow (`renderer.js`)

A module-level `conversationId` (initially `null`) tracks the current chat session, and a
module-level `pendingAttachments` array (`{path, filename}`) tracks staged, unsent attachments
(see [Attachments](#attachments-rendererjs)).

On `composer` submit:

1. Read and trim `input.value`; if empty, do nothing (no request is sent).
2. Take `pendingAttachments` for this send and clear it (re-rendering the now-empty chip row)
   before anything else, so a slow request does not let the user attach the same files twice.
3. Append the user's entry (`You` gutter with the current HH:MM), clear the input, disable the
   send button, and append the thinking entry via `addThinking()` (see
   [Thinking stage](#thinking-stage)).
4. `POST` to `${backendUrl}/chat` with `{ "message": text, "conversation_id": conversationId,
   "attachments": <the paths taken in step 2> }`.
5. On a non-OK HTTP status, throw into the `catch`.
6. On success: set `conversationId = data.conversation_id`, stop the thinking timer and render
   the answer into the same entry (paragraphs, citations, and a sources manifest, see
   [Answers, citations, and sources](#answers-citations-and-sources)), then, below the answer,
   call `addAttachmentResults(data.attachments)` if non-empty and `addFactCard()` if
   `data.pending_fact` is non-`null`. If this request started a new conversation and
   `data.title` is non-`null`, call `prependHistoryItem()` (see [Conversation
   history](#conversation-history)).
7. On any error: remove the thinking entry, append `"Could not reach the backend: <message>"` as
   an `Error` entry.
8. Either way, re-enable the send button and refocus the input.

## Ledger entries (`addEntry`, `addMessage`)

Every stream item goes through `buildEntry(gutterHtml, options)` / `addEntry()`: a `.entry` grid
with a right-aligned `.entry-gutter` and a `.entry-body` carrying the shared vertical rule.
`addMessage(text, role)` maps the old roles onto entries:

- **`"user"`**: an `.entry-you` (accent rule segment) with the `You` gutter label; live sends
  add a `g-time` timestamp, replays do not. Body is a `.q-text` paragraph (pre-wrap).
- **`"assistant"`**: delegates to `addAssistantEntry(text, [])`.
- **`"system"`** / **`"error"`**: a `System` (dim mono) or `Error` (red mono) tagged entry with
  a one-line body, built by `buildSystemEntry()` (also used standalone when a decision card
  resolves in place).

## Thinking stage

`addThinking()` appends an `Assistant` entry whose body is a `.pending` row: a 12px CSS arc
spinner, the mono label `thinking`, and a live elapsed counter (`0s`, `1s`, ...) ticking every
250ms. It returns `{entry, body, timer}`; the submit handler stops the timer and replaces the
body content with the answer in place (so the ledger position is stable), or removes the entry
on error. There is no token counter: `/chat` is a single non-streaming request, so there are no
live token counts to show, and the app does not fake numbers.

## Answers, citations, and sources

`addAssistantEntry(text, sources)` (used for both live answers and replayed history) renders the
answer as paragraphs (split on blank lines, pre-wrap within a paragraph) via
`renderAnswerInto()`. The chat prompt asks the model to cite context entries as `[n]`, so `[n]`
markers whose number maps to a retrieved source (1-based, within `sources.length`) become
superscript teal `.cite` spans; out-of-range brackets stay literal text.

If `sources` is non-empty, `buildSources()` appends a collapsible `Sources` manifest (no
separator line above it): a toggle row with an outline chevron, the label, and a count badge,
then one mono `.src` row per source with its index (teal), document path, section (dimmer), and
a `project_name` tag on the right edge when present. Clicks are delegated on `#messages`:
the toggle collapses/expands (chevron rotates), and clicking a citation expands the manifest and
flashes the matching row (`.flash`, accent-dim background for 900ms).

## Attachments (`renderer.js`)

A file or folder reaches the composer the same way regardless of source, and neither reaches the
backend until the message carrying it is sent (see [chat
attachments](../architecture/memory-model.mdx#chat-attachments)):

- **File picker**: clicking `#attach` calls `window.openproject.selectPaths()` (native file/folder
  dialog, multi-selection) and passes the result to `stagePaths()`.
- **Drag and drop**: `dragenter`/`dragover` on `document` call `preventDefault()` and add a
  `dragover` class to `<body>`, which turns the composer border accent as the drop affordance;
  `dragleave`/`drop` remove it. On `drop`, each `File` entry is resolved to an absolute path via
  `getPathForFile()` and the paths are passed to `stagePaths()`.

`stagePaths(paths)` calls `window.openproject.listFiles(path)` for every path (a no-op flatten for
a file, a recursive walk for a folder) and pushes every returned file not already staged onto
`pendingAttachments`, then calls `renderPendingAttachments()`. If the combined result is empty
(a folder with nothing supported inside it, or an unsupported single file), it appends an
`Error` entry instead.

`renderPendingAttachments()` clears and re-renders `#pending-attachments`, one file chip per
staged attachment. File attachments always render as file chips (`buildFileChip()`: a document
SVG icon plus the mono filename); staged ones additionally carry an `×` remove control that
splices them out of `pendingAttachments`. The staged row hides itself via CSS `:empty`.

### Sending and attachment results

Sent attachments (see [Chat flow](#chat-flow-rendererjs) step 4) come back in `data.attachments`,
handled by `addAttachmentResults(attachments)`:

1. Any entry with `status === "failed"` gets an `Error` entry via
   `addMessage("<filename>: <error>", "error")`.
2. The remaining (successful) entries are rendered inside one `Memory`-tagged entry
   (`.ledger-col` body): one `.ledger-line` per attachment with its file chip, a flexible
   spacer, and a quiet "save to memory" action button (`.act`).
3. If there is more than one successful attachment, a right-aligned "save all to memory" button
   is appended below the rows. Clicking it disables itself ("saving all…"), then `await`s
   `promoteAttachment()` (below) for each row in turn, skipping any row whose button is no
   longer in the DOM (already saved individually), and removes itself once done.

`promoteAttachment(attachment, buttonEl)`: disables `buttonEl` ("saving…"), `POST`s to
`${backendUrl}/documents/${attachment.document_id}/promote`. On success, replaces the button
with an amber note, `"saved to memory"` or `"saved, project unclear"` (`project_resolution ===
"ambiguous"`), calling `loadPendingResolutions()` in the latter case so the new pending card
appears immediately. On error, re-enables the button and appends an `Error` entry; used both
for a single "save to memory" click and, sequentially, for "save all to memory".

## Pagination (`renderer.js`)

`GET /conversations`, `GET /projects`, `GET /project-resolutions`, and `GET /facts/pending` are
paginated (see [Backend](./backend.mdx#api-endpoints)), returning `{ "items": [...], "total":
<count> }` rather than a bare array. `fetchAllPages(path)` hides this: it `GET`s `path` with
`limit=100` and an increasing `offset`, accumulating `items` across requests until `offset >=
total` or a page comes back empty.

## Ambiguous project resolution (`renderer.js`)

`loadPendingResolutions()` runs on startup, after resolving a card (see below), and after a [chat
attachment promotion](#sending-and-attachment-results) comes back `"ambiguous"`. It fetches
pending resolutions via `fetchAllPages("/project-resolutions")`; if any exist, it also fetches
all projects via `fetchProjects()` and calls `addResolutionCard(resolution, projects)` for each.
On a non-OK response or network error, it appends a `"Could not load pending project resolutions:
<message>"` error entry.

`addResolutionCard(resolution, projects)` appends a `Resolve`-tagged entry whose body is an
amber decision card (`.decision`, all its accent elements amber, including the primary button):

- A title, `Which project does <filename> belong to?`, with the filename in inline code style.
- A subtitle repeating the candidate: `Candidate "<candidate_name>" (<candidate_description>).
  The document matches more than one known project.` (the parenthetical is omitted if the
  description is `null`).
- An **"Attach to existing"** group with one radio row (`.opt`) per existing project, showing
  just the project name (the `/projects` payload carries no document counts, so no meta column
  is rendered).
- An **"Or create new"** group with a radio row holding an editable text input, pre-filled with
  `candidate_name`. Focusing the input selects the "create new" option automatically.
- A footer with **dismiss** (removes the entry without POSTing) and **attach**.

Radio selection is managed client-side by `updateSelection(projectId, newMode)`, which toggles
the `sel` class across the option rows (the amber fill and radio dot are pure CSS). The initial
selection is the first existing project, or "create new" if there are no existing projects.

`resolveProjectResolution(resolution, projectId, entryEl, projectName)` disables the card's
buttons, then `POST`s to `${backendUrl}/project-resolutions/<id>/resolve` with `{ "project_id":
projectId }` (`null` creates a new project). On success, the whole entry is replaced in place by
a `System` entry, `<filename> attached to project "<name>"`, then all remaining resolution
cards are removed from the DOM and `loadPendingResolutions()` runs again so they re-render with
an up-to-date project list. On error, appends an `Error` entry and re-enables the buttons.

## Fact confirmation (`renderer.js`)

`loadPendingFacts()` runs once on startup, alongside `loadPendingResolutions()`. It fetches
pending facts via `fetchAllPages("/facts/pending")` and calls `addFactCard(fact)` for each. On a
non-OK response or network error, it appends a `"Could not load pending facts: <message>"` error
entry.

`addFactCard(fact)` appends a `Fact`-tagged entry whose body is an amber decision card:

- A title: `Remember this?`
- The fact statement as a bordered mono line (`.fact-line`): the subject, then
  `<predicate>: <object>` in a dimmer tier, separated by a gap (no dots).
- A footer with **discard** and **remember**.

`resolveFactCard(fact, confirm, entryEl)` disables the buttons, then `POST`s to
`${backendUrl}/facts/<id>/resolve` with `{ "confirm": confirm }`. On success, the entry is
replaced in place by a `System` entry, `"fact recorded"` or `"fact discarded"`. On error,
appends an `Error` entry and re-enables the buttons.

A fact card for the current turn, if `data.pending_fact` is non-`null`, is also added inline
from the `/chat` response (see [Chat flow](#chat-flow-rendererjs) step 6), without waiting for
the next startup.

## What is not here yet

- No backend process management: the FastAPI backend must already be running (on `localhost:8000`
  by default) before `npm start`. The Electron app does not start, stop, or monitor it.
- The active conversation is not auto-resumed across app restarts: `npm start` always opens on a
  clean composer with `conversationId = null`, even if a conversation was open when the app was
  last closed. Past conversations are reachable from [conversation
  history](#conversation-history), but only via an explicit click.
- No conversation management beyond browsing: no rename, no delete, no reordering by newest
  activity after a resumed conversation gets a new reply (see [Conversation
  history](#conversation-history)).
- Answers render as plain paragraphs with `[n]` citations; the model's occasional Markdown
  (bold, lists) is shown literally, not formatted.
- Per-message timestamps exist only for live sends (the current wall clock); the conversations
  API does not return message times, so replayed entries show none.
- The [Settings modal](#settings-modal)'s YouTrack and Slack sections are still UI scaffolding:
  they hold no real config, and nothing persists. `backendUrl` itself is set once at launch from
  `OPENPROJECT_BACKEND_URL`, not editable from within the app.
