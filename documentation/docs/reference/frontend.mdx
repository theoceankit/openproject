# Frontend

The frontend is the Electron app in `frontend/src/`. It has no build step: `npm start` runs
`electron .`, which loads the files in `src/` directly.

## Tech stack

The renderer is a plain web page with no bundler or framework:

- **Tailwind CSS** loaded from CDN, with a custom config in `index.html` that extends the color
  palette with Material Design color tokens (`surface`, `primary`, `on-surface`, etc.) and custom
  spacing and typography scales.
- **Material Symbols** (outlined variable font, from Google Fonts) for iconography.
- **Geist** (400/500/600) for UI and body text; **JetBrains Mono** (400/500) for code labels and
  status text.
- Custom styles in `style.css`: scrollbar styling, the `animate-spin` keyframe used by the
  spinner, and `font-variation-settings` for Material Symbols.

## Processes

- **Main process** (`main.js`): on `app.whenReady()`, creates a single 1280x820 `BrowserWindow`
  (menu bar hidden, `backgroundColor: "#0e0e0e"` to match the dark theme and avoid edge artifacts
  on resize) with `preload: preload.js`, `contextIsolation: true`, `nodeIntegration: false`, loads
  `index.html`, and sets the zoom factor to 1.15 once it finishes loading. Quits when all windows
  are closed, except on macOS (`process.platform !==
  "darwin"`). Also registers two `ipcMain.handle` handlers:
  - `documents:select-paths`: opens a native dialog (`dialog.showOpenDialog` with `properties:
    ["openFile", "openDirectory", "multiSelections"]`) and returns the chosen paths, or `[]` if
    the dialog is canceled.
  - `documents:list-files`: given a file or folder path, returns the flat list of supported files
    it resolves to (`listSupportedFiles()`, a recursive walk filtered to `.md`/`.mdx`/`.pdf`,
    mirroring the backend's `discover_files()`/`SUPPORTED_EXTENSIONS`) or `[]` if the path is
    invalid; this is how a dropped or picked folder becomes one chip per file, see
    [Attachments](#attachments-rendererjs).
- **Preload script** (`preload.js`): calls `contextBridge.exposeInMainWorld("openproject", {
  ... })`, exposing to the renderer:
  - `backendUrl`: the `OPENPROJECT_BACKEND_URL` environment variable, or `"http://localhost:8000"`
    if unset.
  - `selectPaths()`: invokes `documents:select-paths` IPC and resolves to the array of chosen
    paths (or `[]` on cancel).
  - `listFiles(path)`: invokes `documents:list-files` IPC for one path.
  - `getPathForFile(file)`: wraps `webUtils.getPathForFile()` to resolve a dropped `File` object
    to its absolute filesystem path.
- **Renderer** (`index.html`, `renderer.js`, `style.css`): the chat UI, running as a regular web
  page with no Node/Electron access beyond what `preload.js` exposes.

## UI structure

`index.html` defines two top-level regions:

- **Sidebar** (`<nav>`, fixed, 260px wide): a column of navigation items (New Chat, History,
  Projects, Settings, Help, Feedback) and a `#sidebar-history` scroll area listing past
  conversations, see [Conversation history](#conversation-history). The active-state indicator is
  a 2px vertical bar on the left edge of the active row. Settings opens a modal, see [Settings
  modal](#settings-modal), rather than navigating anywhere.
- **Main canvas** (`<main>`, filling the remaining width): a 48px `<header>`, a `#chat` scroll
  area, and a floating footer composer. The `#messages` div inside `#chat` is constrained to
  `max-w-[800px]` and centered.

The **composer** (`<form id="composer">`) is a `flex flex-col` of two rows:
- `#pending-attachments` (`<div>`, hidden when empty): a wrapping row of chips for staged
  attachments, see [Attachments](#attachments-rendererjs).
- The input row: `#attach` (`<button>`, opens the file/folder picker), `#message` (`<textarea>`,
  auto-growing), and a submit button with an `arrow_upward` icon.

## Startup

On DOM ready, the renderer runs three calls in parallel:

- `loadPendingResolutions()`: fetches and renders any `ProjectResolution` rows left pending from
  a previous session (see [Ambiguous project resolution](#ambiguous-project-resolution-rendererjs)).
- `loadPendingFacts()`: fetches and renders any pending fact confirmations (see
  [Fact confirmation](#fact-confirmation-rendererjs)).
- `loadConversationHistory()`: fetches and renders the sidebar's conversation list (see
  [Conversation history](#conversation-history)).

## New Chat

Clicking `#new-chat-btn` (the "New Chat" sidebar item) runs `resetConversationView()`: clears
`messages.innerHTML`, resets `conversationId` to `null` and `pendingAttachments` to `[]`, and
clears the active highlight from `#sidebar-history` (`setActiveHistoryItem(null)`). The backend
keeps the old conversation's rows, but the renderer no longer sends their `conversation_id`, so
replies start a fresh thread; the app never auto-resumes the last conversation on launch, only on
an explicit click in the history list.

## Conversation history

`loadConversationHistory()` fetches [`GET /conversations`](./backend.mdx#get-conversations) (via
the same `fetchAllPages()` helper used for pending resolutions/facts/projects) and replaces
`#sidebar-history`'s content with one `.history-item` button per conversation, in the order the
backend returns them (most recently active first). A `.history-item` shows the conversation's
`title` (falling back to `"New conversation"` if still `null`) and its `preview`, both truncated
with CSS `truncate`. If the list is empty, `renderHistoryList()` restores
`sidebarHistoryDefaultHTML` (the same placeholder markup used by [Clear
database](#clear-database)) instead of rendering nothing.

Clicking a `.history-item` calls `openConversation(id, itemEl)`: fetches [`GET
/conversations/{id}`](./backend.mdx#get-conversationsid), clears `messages.innerHTML`, sets
`conversationId = data.id` and `pendingAttachments = []`, renders one system-style message
(`addMessage(..., "system")`) listing any attachment filenames if `data.attachments` is
non-empty, then replays `data.messages` in order via `addMessage()` (and `addSources()` for
assistant messages), and finally highlights the clicked item via `setActiveHistoryItem(itemEl)`.
From this point, `conversationId` is set, so the next composer submit continues this conversation
rather than starting a new one.

`setActiveHistoryItem(el)` toggles the active styling (background + `text-primary` title) on
`el` and removes it from every other `.history-item`; passing `null` clears the highlight
entirely (used by [New Chat](#new-chat) and [Clear database](#clear-database)).

After a successful `POST /chat` response for a brand-new conversation (`conversationId` was
`null` when the request was sent) that came back with a non-null `title`, the chat flow (below)
calls `prependHistoryItem()` to insert a new `.history-item` at the top of the list and mark it
active, without waiting for the next `loadConversationHistory()`. This is the only case the
sidebar list updates live: continuing an existing conversation does not move or refresh its
existing entry (its `preview`/`updated_at` go stale until the app is next restarted or the list is
otherwise reloaded).

## Settings modal

Clicking the sidebar's "Settings" button opens `#settings-modal`, a two-pane dialog styled like
the rest of the app (Obsidian/Discord-style: a left-hand section list, a right-hand content pane).
`setActiveSettingsSection(section)` toggles `.hidden` on the matching `[data-settings-panel]` and
the active-state styling on the matching `.settings-nav-btn`; the modal always opens back on the
Models section. Five sections: **Models**, **YouTrack**, **Slack**, **Statistics**, **Database**.

Models and Database are wired to real backend calls; YouTrack, Slack, and Statistics remain UI
scaffolding with no persistence, static placeholder data standing in for a future settings API:

- **Models**: a default-model dropdown plus per-task overrides (Chat, Extraction, Embeddings,
  Orchestration). A native `<select>`'s open list can't be restyled to match the app's dark
  theme, so these are a custom combobox (`.model-dropdown`): a trigger button plus an
  absolutely-positioned `.model-dropdown-menu`. `initModelDropdowns()` runs once at script load
  (not on every modal open, since the elements exist in the DOM regardless of the modal's
  `hidden` state and re-running it would double-attach listeners): it fetches [`GET
  /settings/models`](./backend.mdx#get-settingsmodels) and, for each `.model-dropdown`, either
  renders it editable (`initEditableModelDropdown()`, options from the response's
  `available_llm_models`, a leading "Use default" option for the four per-task ones, none for
  Default model) or, for the one `data-task="embeddings"` container, calls
  `renderEmbeddingsRow()` to show `embeddings_model` as plain disabled text with no menu (see
  [Selection and configuration](./llm-integration.mdx#selection-and-configuration) for why the
  embedding model isn't editable from here). If the initial fetch fails, every trigger is
  disabled, its value set to "Unavailable", and the error is shown in `#model-settings-error`.
  Selecting an option autosaves: `setModelDropdownValue()` updates the UI optimistically,
  `#model-settings-status` shows "Saving…" then "Saved" (auto-clearing after 2s), and a `PATCH
  /settings/models` failure reverts the dropdown to its previous value and shows the error in
  `#model-settings-error` instead. `closeAllModelDropdowns()` runs on any document click, so
  opening one closes the others and clicking outside closes all of them. There is no cloud
  provider option group: none is implemented yet (see [model
  providers](../architecture/model-providers.mdx#hosted-providers)).
- **YouTrack** / **Slack**: static forms (URL/token/default-project; workspace/bot-token/signing-
  secret/default-channel) with a "Not connected" status pill and a password-style secret input
  with a show/hide toggle (`[data-toggle-password]`). "Test connection" and "Save" render but do
  nothing.
- **Statistics**: plain label/value rows (not cards), grouped under "Corpus" (Projects, Documents,
  Chunks, Conversations) and "Usage" (Model calls, Tokens used) headings, with a caption noting the
  numbers are illustrative since the backend does not track usage metrics.
- **Database**: see [Clear database](#clear-database) below.

## Clear database

The Settings modal's Database section holds a "Clear database" button (styled with the app's error
color tokens) that opens a confirmation modal warning that the action permanently deletes all
projects, documents, and conversation history. Once Confirm is clicked, both Cancel and the backdrop are
disabled (a module-level `clearDatabaseInFlight` flag) until the request settles, since the
backend call is already dispatched by then and cannot actually be cancelled. On success, it calls
[`POST /admin/reset`](./backend.mdx#post-adminreset), then runs the same `resetConversationView()`
used by [New Chat](#new-chat) and additionally restores `#sidebar-history` to the `innerHTML` it
had at module load (captured once into `sidebarHistoryDefaultHTML`, rather than a second hardcoded
copy of the placeholder markup). On failure, the error is shown inline in the modal
(`#clear-database-error`) and the modal stays open so the user can retry.

## Chat flow (`renderer.js`)

A module-level `conversationId` (initially `null`) tracks the current chat session, and a
module-level `pendingAttachments` array (`{path, filename}`) tracks staged, unsent attachments
(see [Attachments](#attachments-rendererjs)).

On `composer` submit:

1. Read and trim `input.value`; if empty, do nothing (no request is sent).
2. Take `pendingAttachments` for this send and clear it (re-rendering the now-empty chip row)
   before anything else, so a slow request does not let the user attach the same files twice.
3. Append the user's message via `addMessage(text, "user")`, clear the input, disable the input
   and send button, and show a loading indicator via `addSpinner("thinking…")`.
4. `POST` to `${backendUrl}/chat` with `{ "message": text, "conversation_id": conversationId,
   "attachments": <the paths taken in step 2> }`.
5. On a non-OK HTTP status, throw into the `catch`.
6. On success: set `conversationId = data.conversation_id`, remove the spinner, if
   `data.attachments` is non-empty call `addAttachmentResults(data.attachments)`, append the
   answer via `addMessage(data.answer, "assistant")`, call `addSources()` on the returned card
   element, and if `data.pending_fact` is non-`null`, call `addFactCard(data.pending_fact)`. If
   this request started a new conversation (`conversationId` was `null` before step 4) and
   `data.title` is non-`null`, call `prependHistoryItem()` (see [Conversation
   history](#conversation-history)).
7. On any error: remove the spinner, append `"Could not reach the backend: <message>"` via
   `addMessage(..., "error")`.
8. Either way, re-enable the input and button and refocus the input.

## Message rendering (`addMessage`)

`addMessage(text, role)` creates and appends a wrapper div to `#messages`, then scrolls `#chat`
to the bottom. The DOM structure differs by role:

- **`"user"`**: `flex justify-end w-full` wrapper, containing a right-aligned card
  (`bg-surface-container-high`, `max-w-[80%]`, `rounded-xl`, `border border-white/5`).
- **`"assistant"`**: `flex flex-col items-start w-full` wrapper, containing a full-width card
  (`bg-surface-container`, `w-full`, `rounded-xl`). `addSources()` appends a sources row to this
  card element.
- **`"error"`**: same wrapper shape as assistant, with a red-tinted card (inline styles:
  `background: rgba(147,0,10,0.15)`, `border: 1px solid rgba(255,180,171,0.2)`).
- **`"system"`**: `flex items-center gap-xs` wrapper, no card background. Renders a small `info`
  Material Symbol and a monospace label in `text-on-surface-variant/40`.

Returns the card element (or wrapper for `"system"`) so the caller can append children.

## Spinner (`addSpinner`)

`addSpinner(text)` appends a flex row with a spinner and a text label, and returns the wrapper so
the caller can remove it when the operation completes.

The spinner is a two-layer structure: a static 20x20 pill (`bg-surface-container-high`,
`border border-white/10`, `rounded-full`) as the background container, with a 12x12 CSS arc
spinner inside it (`border-2 border-white/20 border-t-white/70`, `rounded-full`,
`animate-spin`). The background pill does not rotate; only the inner arc does, keeping the
rotation center geometrically stable.

## Sources (`addSources`)

`addSources(container, sources)` appends a "Sources" row to the given container element (the
assistant card returned by `addMessage`). It renders a muted `SOURCES` label followed by one
badge per source, each showing `[project_name] path (section)`, `[project_name] path`,
`path (section)`, or just `path`, depending on which fields are non-null. Does nothing if
`sources` is empty or absent.

## Attachments (`renderer.js`)

A file or folder reaches the composer the same way regardless of source, and neither reaches the
backend until the message carrying it is sent (see [chat
attachments](../architecture/memory-model.mdx#chat-attachments)):

- **File picker**: clicking `#attach` calls `window.openproject.selectPaths()` (native file/folder
  dialog, multi-selection) and passes the result to `stagePaths()`.
- **Drag and drop**: `dragenter`/`dragover` on `document` call `preventDefault()` and add a
  `dragover` CSS class to `#chat` for a visual drop highlight; `dragleave`/`drop` remove it. On
  `drop`, each `File` entry is resolved to an absolute path via `getPathForFile()` and the paths
  are passed to `stagePaths()`.

`stagePaths(paths)` calls `window.openproject.listFiles(path)` for every path (a no-op flatten for
a file, a recursive walk for a folder) and pushes every returned file not already staged onto
`pendingAttachments`, then calls `renderPendingAttachments()`. If the combined result is empty
(a folder with nothing supported inside it, or an unsupported single file), it appends an
`"error"` message instead. There is no other branch: a folder does not do anything a file does not
also do, just as one attachment does not do anything several do not also do; see
[improvements backlog](../contributing/improvements-backlog.mdx) history for the older,
file/folder-branching version this replaced.

`renderPendingAttachments()` clears and re-renders `#pending-attachments`, one chip per staged
attachment (filename plus a `close` icon button that splices it out of `pendingAttachments` and
re-renders), toggling the row's `hidden` class based on whether the array is empty.

### Sending and attachment results

Sent attachments (see [Chat flow](#chat-flow-rendererjs) step 4) come back in `data.attachments`,
handled by `addAttachmentResults(attachments)`:

1. Any entry with `status === "failed"` gets an `"error"` message via
   `addMessage("<filename>: <error>", "error")`.
2. The remaining (successful) entries are rendered as rows inside one container appended to
   `#messages`: `addAttachmentResultRow(container, attachment)` builds a pill (a colored dot,
   `"<filename> · attached to this conversation, not saved"`, and a **Save to memory** button) and
   returns `{attachment, buttonEl, labelEl}`.
3. If there is more than one successful attachment, a **Save all to memory** button is inserted
   before the rows. Clicking it disables itself, then `await`s `promoteAttachment()` (below) for
   each row in turn, skipping any row whose button is no longer in the DOM (already saved
   individually), and removes itself once done.

`promoteAttachment(attachment, buttonEl, labelEl)`: disables `buttonEl` and sets its text to
"Saving…", `POST`s to `${backendUrl}/documents/${attachment.document_id}/promote`. On success,
removes `buttonEl` and updates `labelEl` to `"<filename> · saved to memory"` or `"<filename> ·
saved, project unclear"` (`project_resolution === "ambiguous"`), calling
`loadPendingResolutions()` in the latter case so the new pending card appears immediately. On
error, re-enables the button and appends an `"error"` message; used both for a single "Save to
memory" click and, sequentially, for "Save all to memory".

## Pagination (`renderer.js`)

`GET /projects`, `GET /project-resolutions`, and `GET /facts/pending` are paginated (see
[Backend](./backend.mdx#api-endpoints)), returning `{ "items": [...], "total": <count> }` rather
than a bare array. `fetchAllPages(path)` hides this: it `GET`s `path` with `limit=100` and an
increasing `offset`, accumulating `items` across requests until `offset >= total` or a page comes
back empty.

## Ambiguous project resolution (`renderer.js`)

`loadPendingResolutions()` runs on startup, after resolving a card (see below), and after a [chat
attachment promotion](#sending-and-attachment-results) comes back `"ambiguous"`. It fetches
pending resolutions via `fetchAllPages("/project-resolutions")`; if any exist, it also fetches
all projects via `fetchProjects()` and calls `addResolutionCard(resolution, projects)` for each.
On a non-OK response or network error, it appends a `"Could not load pending project resolutions:
<message>"` error message.

`addResolutionCard(resolution, projects)` appends a card with:

- An amber indicator row ("Your decision needed") above the card.
- A title: `Which project does <filename> belong to?`
- A subtitle: `Candidate: "<candidate_name>" -- <candidate_description>` (the description is
  omitted if `null`).
- An **"Add to existing"** section with one radio-button row per existing project, all initially
  deselected except the first.
- A **"Create new project"** section with a radio-button row and an editable text input,
  pre-filled with `candidate_name`. Focusing the input selects the "create new" option
  automatically.
- **Cancel** (removes the card without POSTing) and **Attach** footer buttons.

Radio selection is managed client-side by `updateSelection(pid, newMode)`, which toggles border
and fill across all option rows. The initial selection is the first existing project, or "create
new" if there are no existing projects.

`resolveProjectResolution(resolution, projectId, cardEl)` disables the card's buttons, then
`POST`s to `${backendUrl}/project-resolutions/<id>/resolve` with `{ "project_id": projectId }`
(`null` creates a new project). On success, removes the resolved card, appends a
`"<document_path>: project resolved"` system message, removes all remaining resolution cards from
the DOM, and calls `loadPendingResolutions()` again so the remaining cards re-render with an
up-to-date project list. On error, appends an error message and re-enables the buttons.

## Fact confirmation (`renderer.js`)

`loadPendingFacts()` runs once on startup, alongside `loadPendingResolutions()`. It fetches
pending facts via `fetchAllPages("/facts/pending")` and calls `addFactCard(fact)` for each. On a
non-OK response or network error, it appends a `"Could not load pending facts: <message>"` error
message.

`addFactCard(fact)` appends a card with:

- A blue indicator row ("Confirm fact") above the card.
- A title: `Remember this?`
- The fact text (`<subject> <predicate>: <object>`) in a monospace block.
- **Discard** and **Remember** buttons.

`resolveFactCard(fact, confirm, cardEl)` disables the buttons, then `POST`s to
`${backendUrl}/facts/<id>/resolve` with `{ "confirm": confirm }`. On success, removes the card
and appends `"Recorded."` (confirm) or `"Discarded."` (reject) as a system message. On error,
appends an error message and re-enables the buttons.

A fact card for the current turn, if `data.pending_fact` is non-`null`, is also added inline
from the `/chat` response (see [Chat flow](#chat-flow-rendererjs) step 5), without waiting for
the next startup.

## What is not here yet

- No backend process management: the FastAPI backend must already be running (on `localhost:8000`
  by default) before `npm start`. The Electron app does not start, stop, or monitor it.
- The active conversation is not auto-resumed across app restarts: `npm start` always opens on a
  clean composer with `conversationId = null`, even if a conversation was open when the app was
  last closed. Past conversations are reachable from [conversation
  history](#conversation-history), but only via an explicit click.
- No conversation management beyond browsing: no rename, no delete, no reordering by newest
  activity after a resumed conversation gets a new reply (see [Conversation
  history](#conversation-history)).
- The [Settings modal](#settings-modal)'s YouTrack, Slack, and Statistics sections are still UI
  scaffolding: they hold no real config, and nothing persists. `backendUrl` itself is set once at
  launch from `OPENPROJECT_BACKEND_URL`, not editable from within the app.
