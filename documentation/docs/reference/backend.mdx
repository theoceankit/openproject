# Backend

The backend is the FastAPI app in `backend/app/`. This page covers its data model, its API, and
how the ingestion, extraction, retrieval, and chat pipelines work. For where each piece lives,
see the [codebase map](../contributing/codebase-map.mdx); for why it is shaped this way, see the
[architecture docs](../architecture/system-architecture.mdx).

## Application setup

`app/main.py` creates the FastAPI app, mounts `CORSMiddleware` (so the Electron renderer's
`file://` origin can call the API), and mounts one router per `app/api/` module: `health`,
`documents`, `projects`, `project_resolutions`, `chat`.

Configuration (`app/core/config.py`) is a `pydantic-settings` `Settings` object, read from
environment variables prefixed `OPENPROJECT_` or from `backend/.env`:

| Setting | Default | Used for |
| --- | --- | --- |
| `database_url` | `postgresql+asyncpg://openproject:openproject@localhost:5432/openproject` | the async SQLAlchemy engine |
| `cors_allow_origins` | `*` | comma-separated origins allowed by `CORSMiddleware` |
| `ollama_host` | `http://localhost:11434` | the Ollama provider |
| `ollama_timeout_seconds` | `120.0` | client-side timeout for Ollama `generate()`/`embed()` calls |
| `llm_model` | `qwen2.5:14b-instruct` | `generate()` calls |
| `embedding_model` | `bge-m3` | `embed()` calls |
| `embedding_dim` | `1024` | the `Chunk.embedding` `pgvector` column type |
| `extraction_max_chars` | `12000` | how much of a document's text is sent for extraction |
| `chat_retrieval_limit` | `5` | how many chunks are retrieved per chat message |
| `chat_attachment_retrieval_limit` | `8` | how many chunks are retrieved from the current conversation's [attachments](#chat-attachments) per chat message |
| `chat_history_max_chars` | `4000` | how much prior conversation is included in chat prompts |
| `log_level` | `INFO` | root logger level |
| `log_dir` | `logs` | directory for log files |
| `log_sql_queries` | `false` | whether SQL statements are logged |
| `log_llm_interactions` | `true` | whether `generate()`/`embed()` calls are logged |
| `log_llm_preview_chars` | `200` | max length of the prompt/response preview logged to the console |

`app/db/session.py` creates a single async engine and `async_sessionmaker`. The `get_db()`
dependency yields one `AsyncSession` per request.

`app/core/logging.setup_logging()` runs at import time, configuring the loggers and log files
under `log_dir`; see [Logging](./logging.mdx).

## Data model

All tables (`app/db/models.py`) use `UUIDPrimaryKeyMixin` (a `uuid4` primary key) and
`TimestampMixin` (`created_at`, `updated_at`).

- **`Project`**: `name`, `description`. The top-level container.
- **`Document`**: `project_id` (nullable FK to `Project`), `path` (absolute, used as the
  dedup key on re-ingestion, with a unique constraint), `doc_type` (`"markdown"` or `"pdf"`),
  `content_hash` (sha256 of the file's bytes), `origin` (`"ingested"` (default) or
  `"attachment"`; see [chat attachments](#chat-attachments)).
- **`Chunk`**: `document_id` (FK), `chunk_index`, `content`, `section` (nullable, a heading path
  for markdown or `"Page N"` for PDF), `embedding` (`pgvector` `Vector(embedding_dim)`, nullable
  at the column level but always set when a chunk is created).
- **`Term`**: `project_id` (nullable), `document_id`, `source_section`, `term`, `definition`.
- **`Team`**: `project_id` (nullable), `document_id`, `source_section`, `name`.
- **`Person`**: `project_id` (nullable), `team_id` (nullable FK to `Team`), `document_id`,
  `source_section`, `name`, `role` (nullable).
- **`Topic`**: `project_id` (nullable), `document_id` (nullable), `source_section`, `name`,
  `description` (nullable). A catch-all for relation endpoints that are not Project/Person/Team.
  `document_id` is `null` for topics created from a chat-stated fact rather than extracted from a
  document.
- **`Relation`**: a generic `(subject_type, subject_id, relation_label, object_type, object_id)`
  triple, plus `document_id` and `source_section`. `subject_id` / `object_id` are UUIDs tagged
  with a type string (`"project"`, `"person"`, `"team"`, `"topic"`); there is no database-level
  foreign key, the type string says which table to look in.
- **`ProjectResolution`**: `document_id`, `candidate_name`, `candidate_description`,
  `candidate_project_ids` (array of UUIDs), `status` (`"pending"` or `"resolved"`),
  `resolved_project_id` (nullable).
- **`Conversation`**: no fields beyond `id` and the timestamps. A chat session; groups `Message`
  rows.
- **`Message`**: `conversation_id` (FK), `role` (`"user"` or `"assistant"`), `content`, `sources`
  (`JSONB`, nullable; for assistant messages, a snapshot of
  `[{"document_path", "section", "project_name"}, ...]` for the chunks cited in that answer).
- **`ConversationAttachment`**: `conversation_id` (FK), `document_id` (FK). A join row linking an
  attachment-origin `Document` to a conversation it is staged for; see [chat
  attachments](#chat-attachments).
- **`Fact`**: an append-only assertion stated in chat, `(subject_type, subject_id, predicate,
  object_type, object_id, value)`, plus `source_type`/`source_id` (currently always
  `"message"`/a `Message.id`) and `status` (`"pending"`, `"confirmed"`, or `"rejected"`, default
  `"pending"`). `subject_id`/`object_id` are tagged UUIDs like `Relation`'s, with no
  database-level foreign key. `object_type`/`object_id` and `value` are both nullable, but not
  both `null`. No unique constraint: restating a fact for the same
  `(subject_type, subject_id, predicate)` inserts a new row rather than replacing the old one, so
  the history is the full set of matching rows. See [chat](#chat) and [chat-derived
  facts](../architecture/memory-model.mdx#chat-derived-facts).

`project_id` is nullable on `Document`, `Term`, `Team`, `Person`, and `Topic` because a document
whose project resolution is "ambiguous" is ingested and extracted before the user confirms which
project it belongs to; see [project resolution](#project-resolution) and
[invariants](./invariants.mdx).

Every foreign key column is indexed (`project_id`, `document_id`, `team_id`,
`conversation_id`, `resolved_project_id`), as are `Relation`/`Fact`'s tagged `subject_id` and
`object_id`, despite those having no database-level foreign key. `Chunk.embedding` has an `hnsw`
index with `vector_cosine_ops`, matching the `cosine_distance` ordering [retrieval](#retrieval)
uses.

## API endpoints

`GET /projects`, `GET /project-resolutions`, and `GET /facts/pending` are paginated: each takes
`limit` (default 50, max 200) and `offset` (default 0) query parameters, and wraps its rows as
`{ "items": [...], "total": <count> }`, `total` being the count of matching rows regardless of
`limit`/`offset`. `app/api/pagination.Page` is the shared response model.

### `GET /health`, `GET /health/db`, `GET /health/model`

Liveness checks. `/health/db` runs `SELECT 1`. `/health/model` calls `provider.generate(...)`
with a trivial prompt and `provider.embed(["health check"])`, and returns the generated text and
the embedding's length, useful to confirm Ollama is reachable and both configured models are
pulled.

### `POST /documents/ingest`

Request:

```json
{ "path": "/absolute/path/to/file/or/folder" }
```

- `404` if the path does not exist.
- `400` if it is a folder (or file) with no `.md`, `.mdx`, or `.pdf` files.
- Otherwise, every matching file is ingested (see [ingestion](#ingestion-pipeline)), and every
  file whose ingestion status is not `"unchanged"` is also run through
  [extraction](#extraction-pipeline).

Response, one entry per discovered file:

```json
[
  {
    "path": "/absolute/path/to/file.md",
    "status": "ingested",
    "chunks": 4,
    "project_id": "f0b6...",
    "project_resolution": "new"
  }
]
```

- `status` is `"ingested"`, `"updated"`, `"unchanged"`, or `"failed"`.
- `project_id` and `project_resolution` are `null` for `"unchanged"` and `"failed"` documents.
- `project_resolution` is `"match"`, `"new"`, or `"ambiguous"` for `"ingested"` /
  `"updated"` documents; `project_id` is `null` when it is `"ambiguous"`.
- `error` is set (and `status` is `"failed"`) when ingestion itself raised for that file (for
  example a non-UTF-8 markdown file, or a PDF `pypdf` cannot parse); it is also set, with `status`
  left as `"ingested"`/`"updated"`, when ingestion succeeded but the extraction step raised (for
  example a connection error to Ollama, or a malformed LLM response). Either way, a failure for
  one file does not stop the request: each file is processed independently and the response
  always has one entry per discovered file, so the caller can tell which files, if any, succeeded.
- Every document this endpoint ingests gets (or keeps) `origin = "ingested"`: if a path was
  previously staged as a [chat attachment](#chat-attachments) (`origin = "attachment"`) and is
  now ingested directly, it is promoted the same way [`POST
  /documents/{id}/promote`](#post-documentsidpromote) would, including running extraction even
  when the file's content is otherwise unchanged (normally skipped for an `"unchanged"` file).

### `POST /documents/{id}/promote`

Moves a [chat attachment](#chat-attachments) into the permanent corpus. `404` if the document does
not exist, `400` if its `origin` is not `"attachment"` (already part of the corpus). On success,
sets `origin = "ingested"`, runs [`extract_document()`](#extraction-pipeline) (the same project
resolution as `POST /documents/ingest`), and returns:

```json
{ "document_id": "...", "project_id": "f0b6...", "project_resolution": "match" }
```

`project_id` is `null` when `project_resolution` is `"ambiguous"`, same as ingestion; a
`ProjectResolution` row is created for later confirmation via [`GET
/project-resolutions`](#get-project-resolutions) exactly as it would be for a directly-ingested
document.

### `GET /projects`

Returns `{ "items": [{ "id": "...", "name": "...", "description": "..." }], "total": <count> }`,
`items` ordered by name.

### `GET /project-resolutions`

Returns pending `ProjectResolution` rows, joined with their document's path, ordered by
`created_at`:

```json
{
  "items": [
    {
      "id": "...",
      "document_id": "...",
      "document_path": "/absolute/path/to/file.md",
      "candidate_name": "Storefront Redesign",
      "candidate_description": "...",
      "candidate_project_ids": ["...", "..."],
      "status": "pending"
    }
  ],
  "total": 1
}
```

Only rows with `status == "pending"` are returned (and counted in `total`); resolved rows drop out
of this list.

### `POST /project-resolutions/{id}/resolve`

Request is `{}`, `{ "project_id": null }`, or `{ "project_id": "<uuid>" }`.

- `{}` / `{ "project_id": null }`: creates a new `Project` from the resolution's
  `candidate_name` and `candidate_description`.
- `{ "project_id": "<uuid>" }`: attaches the document to that existing project. `404` if it does
  not exist.
- `400` for an invalid UUID, `404` if the resolution id does not exist, `400` if it is not
  `"pending"`.

On success, runs [`apply_resolution`](#project-resolution) and returns
`{ "project_id": "...", "status": "resolved" }`.

### `POST /chat`

Request: `{ "message": "...", "conversation_id": "...", "attachments": ["/absolute/path", ...] }`.
`conversation_id` is optional; omit it (or pass `null`) to start a new conversation. `attachments`
is optional, defaulting to `[]`; each path is staged as a [chat attachment](#chat-attachments) for
this conversation before the message is answered (see [chat](#chat) step 2a). Response:

```json
{
  "conversation_id": "...",
  "answer": "...",
  "sources": [{ "document_path": "...", "section": "...", "project_name": "...", "is_attachment": false }],
  "pending_fact": { "id": "...", "subject": "...", "predicate": "...", "object": "..." },
  "attachments": [
    { "path": "...", "filename": "...", "status": "ingested", "chunks": 2, "document_id": "...", "error": null }
  ]
}
```

- `400` if `conversation_id` is not a valid UUID.
- `404` if `conversation_id` does not refer to an existing `Conversation`.

`sources` has at most `chat_retrieval_limit` corpus entries plus up to
`chat_attachment_retrieval_limit` attachment entries (`is_attachment: true`), in the same order as
the numbered context given to the LLM. `pending_fact` is `null` unless this turn recorded a new
`Fact` (see [chat](#chat) step 4b and [Fact resolution](#fact-resolution)). `attachments` mirrors
`attachments` from the request, one entry per path, in the same shape [ingestion of an
attachment](#chat-attachments) returns (`status` is `"ingested"`, `"updated"`, `"unchanged"`, or
`"failed"`, matching `POST /documents/ingest`'s statuses); empty if the request had none.

### `GET /facts/pending`

Returns `Fact` rows with `status == "pending"`, ordered by `created_at`, each described via
[`describe_fact()`](#chat-memory-facts):

```json
{ "items": [{ "id": "...", "subject": "...", "predicate": "...", "object": "..." }], "total": 1 }
```

### `POST /facts/{id}/resolve`

Request: `{ "confirm": true }` or `{ "confirm": false }`.

- `400` for an invalid UUID, `404` if the fact id does not exist, `400` if its `status` is not
  `"pending"`.
- On success, runs [`resolve_fact()`](#chat-memory-facts) (`status` becomes `"confirmed"` or
  `"rejected"`), commits, and returns `{ "id": "...", "status": "confirmed" | "rejected" }`. See
  [Fact resolution](#fact-resolution).

## Ingestion pipeline

`app/ingestion/pipeline.py`, called once per discovered file:

1. `discover_files()` scans a path: if it is a file, returns it if its extension is `.md`,
   `.mdx`, or `.pdf`; if it is a folder, recursively returns all matching files, sorted.
2. `compute_content_hash()` is the sha256 of the file's raw bytes.
3. The file's resolved absolute path is looked up against existing `Document.path` rows.
   - If found and `content_hash` matches: return `{"status": "unchanged", "chunks": 0}` and do
     nothing else.
   - Otherwise, parse the file: `app/ingestion/parsers.parse_markdown()` for `.md`/`.mdx` splits
     on headings, producing one section per heading with a `" > "`-joined nested heading path
     (or `None` for content before the first heading); `parse_pdf()` produces one section per
     page, titled `"Page N"`.
4. If the document existed, its `content_hash` is updated and its existing `Chunk` rows are
   deleted (status `"updated"`); otherwise a new `Document` row is created (status
   `"ingested"`).
5. Each section is embedded as `"{section}\n\n{content}"` (or just `content` when `section` is
   `None`), so the heading path contributes to the embedding without being stored in `Chunk.content`.
   All sections of a document are embedded in a single `provider.embed()` call, and one `Chunk` is
   created per section (`chunk_index`, `content`, `section`, `embedding`).
6. Commit.

Files are processed one at a time, in the sorted order `discover_files()` returns; there is no
concurrency across files within one `/documents/ingest` request. `ingest_path()` isolates each
file's failures: if `ingest_file()` raises (a decode error, a PDF `pypdf` cannot parse), the
session is rolled back, that file's result becomes `{"status": "failed", "chunks": 0, "error":
<message>}`, and the loop continues with the next file, rather than the exception propagating out
of the request and leaving every later file untried.

## Chat attachments

`app/ingestion/pipeline.py`'s `ingest_attachment()` (one file) and `ingest_attachments()` (a
list, isolating per-file failures the same way `ingest_path()` does) stage a file as ephemeral,
conversation-scoped context (see [memory model](../architecture/memory-model.mdx#chat-attachments)),
called from `POST /chat` (step 2a below) once per path in the request's `attachments`.

Both share `_write_document_chunks()` with the plain ingestion path (parse, hash-check, chunk,
embed), parameterized by `origin`: a new `Document` created here gets `origin = "attachment"`
instead of `"ingested"`, and **no extraction runs** (unlike `POST /documents/ingest`'s per-file
loop). `ingest_attachment()` additionally:

- Rejects (`status: "failed"`) a path whose extension is not `.md`/`.mdx`/`.pdf`, or that does not
  exist, since unlike `discover_files()` these paths are not pre-filtered.
- After writing chunks, links the `Document` to the conversation via a `ConversationAttachment`
  row, reusing an existing link for that `(conversation_id, document_id)` pair instead of
  duplicating it (so re-attaching the same file to the same conversation, or promoting it later
  and re-attaching, does not create redundant rows).
- Returns `{"path", "filename", "status", "chunks", "document_id", "error"}`, the shape returned
  in `POST /chat`'s response `attachments` list.

`app/retrieval/search.search_attachment_chunks(db, provider, conversation_id, query, limit)` is
the read side: like `search_chunks()`, but joined through `ConversationAttachment` and filtered to
one conversation instead of filtering attachment-origin documents out. See [retrieval](#retrieval)
for how the two are combined in a chat turn, and [`POST
/documents/{id}/promote`](#post-documentsidpromote) for moving an attachment into the permanent
corpus.

## Extraction pipeline

`app/extraction/pipeline.extract_document()`, called for each non-`"unchanged"` document:

1. Load the document's `Chunk` rows, ordered by `chunk_index`.
2. `_truncate_sections()` keeps `(section, content)` pairs in order until their combined length
   reaches `extraction_max_chars` (default 12000 characters), truncating the last section if it
   would overflow. Later sections are dropped entirely if the limit is already reached.
3. `provider.generate()` is called with `EXTRACTION_SYSTEM_PROMPT`, a prompt built by
   `build_extraction_prompt()` (the document path, then each section as `## <section path>`
   followed by its content), and `format=ExtractionResult.model_json_schema()`. The response is
   parsed as `ExtractionResult` (`project`, `people`, `terms`, `teams`, `relations`); see [LLM
   integration](./llm-integration.mdx) for the prompts and schema.
4. `_clear_prior_extraction()` deletes all `Relation`, `Person`, `Team`, `Term`, and `Topic` rows
   for this document, so a re-extraction always fully replaces them rather than merging.
5. [`resolve_project()`](#project-resolution) resolves `result.project` against existing
   projects, passing the document's current `project_id` (still the prior resolution's value at
   this point, since `_clear_prior_extraction()` does not touch it) as `existing_project_id`;
   `document.project_id` is then set to the result (or left `null` if `"ambiguous"`).
6. `Team` rows are created from `result.teams`. `Person` rows are created from `result.people`,
   with `team_id` set by matching each team's `members` (by name) to a person of that name.
   `Term` rows are created from `result.terms`. All of these get `project_id =
   document.project_id` (which may be `null`).
7. For each `result.relations` triple, [`resolve_entity_ref()`](#project-resolution) resolves
   the subject and object names to `(type, id)` pairs, and a `Relation` row is created.
8. Commit.

`POST /documents/ingest`'s per-document loop catches and isolates a raise from
`extract_document()` the same way: the session is rolled back and the document's `error` field is
set, but its ingestion `status` (already `"ingested"` or `"updated"`) is left as is, and the loop
moves on to the next document.

## Project resolution

`app/extraction/resolution.py`:

### `resolve_project(db, provider, document_id, candidate, existing_project_id=None)`

- If `existing_project_id` is given and still refers to an existing `Project`, it is returned
  immediately as outcome `"match"`, with no LLM call: a document's project assignment is sticky
  across re-extraction, rather than being re-derived (and risking a different, non-deterministic
  outcome from the same LLM resolution call for what is otherwise the same candidate and existing
  projects).
- Otherwise, if there are **no existing `Project` rows**, a new `Project` is created from
  `candidate` immediately, outcome `"new"`, with no LLM call.
- Otherwise, `provider.generate()` is called with `RESOLUTION_SYSTEM_PROMPT` and a prompt
  containing the candidate and all existing projects' `{id, name, description}`, with
  `format=ProjectResolutionResult.model_json_schema()`.
  - `"match"`: returns the existing project's id, provided it is one of the existing projects
    passed into the prompt; if the LLM returns an id that does not match any of them (a
    hallucinated or stale id), the outcome is downgraded to `"ambiguous"` with every existing
    project as a candidate, instead of being used as is.
  - `"new"`: creates a new `Project` from `candidate`.
  - `"ambiguous"`: creates a `ProjectResolution` row (`status="pending"`,
    `candidate_project_ids` from the LLM's `candidate_ids`), and returns `(None, "ambiguous")`.

### `resolve_entity_ref(db, project_id, name, local_entities, document_id)`

Resolves a relation's subject or object name to `(type, id)`, in this order:

1. `local_entities`, a dict built earlier in the same extraction pass (the resolved project's
   name, and this document's new `Team` and `Person` names).
2. An existing `Person` or `Team` row with that `name` and matching `project_id`.
3. An existing `Project` with that `name` (any project, not scoped).
4. An existing `Topic` with that `name`, scoped to `project_id` (or to `project_id IS NULL` if
   `project_id` is `None`).
5. Otherwise, creates a new `Topic` with that `name`, scoped to `project_id` and `document_id`.

Each of steps 2-4 picks one row deterministically (ordered by `created_at`, then `id`) rather than
raising when more than one row matches, for example two same-named people in the same project, or
two same-named Topics. Resolving between several same-named entities properly (rather than just
picking one) is open, see the [entity matching open
question](../architecture/memory-model.mdx#open-questions--risks).

`document_id` is nullable: [`record_fact`](#chat-memory-facts) calls this with `document_id=None`
for facts stated in chat, so a `Topic` created this way has no source document.

### `apply_resolution(db, resolution, project_id)`

Called from `POST /project-resolutions/{id}/resolve`:

- If `project_id` is `None`, looks for an existing `Project` whose `name` matches
  `resolution.candidate_name` (case-insensitive). Reuses it if found; otherwise creates a new
  `Project` from `resolution.candidate_name` / `candidate_description`.
- Sets `document.project_id` to the resolved project.
- For `Term`, `Team`, `Person`, and `Topic`, updates every row for this `document_id` where
  `project_id IS NULL` to the resolved `project_id` (these are the rows created while the
  resolution was pending).
- Sets `resolution.status = "resolved"` and `resolution.resolved_project_id`.

## Retrieval

`app/retrieval/search.search_chunks(db, provider, query, limit)`:

1. `provider.embed([query])` produces the query embedding.
2. `SELECT` joins `Chunk` to `Document`, left-joins `Project` on `Document.project_id` (a document
   can be unscoped), filters to `Document.origin != "attachment"`, orders by
   `Chunk.embedding.cosine_distance(query_embedding)` (pgvector), and takes `limit` rows.
3. Returns `RetrievedChunk(document_path, section, content, project_name, is_attachment)` for
   each; `project_name` is `None` for an unscoped document, `is_attachment` is always `False` here.

This is global across all documents and projects in the permanent corpus: there is no `WHERE
project_id = ...` filter (though attachment-origin documents are excluded, see [chat
attachments](#chat-attachments)). `project_name` on each result lets the caller tell which
project a source came from, even though retrieval itself is not scoped to one.

`search_attachment_chunks(db, provider, conversation_id, query, limit)` is the equivalent query
scoped the other way: joined through `ConversationAttachment` and filtered to one
`conversation_id`, with no `Document.origin` filter and no `Project` join (attachments are
unscoped until promoted). Returns the same `RetrievedChunk` shape with `is_attachment = True`.
Used by [chat](#chat) alongside `search_chunks()`, never on its own.

## Chat

`app/chat/pipeline.answer_question(db, provider, message, conversation=None, attachment_paths=None)`:

1. If `conversation` is `None`, create and flush a new `Conversation` (no history). Otherwise,
   load that conversation's `Message` rows ordered by `created_at`.
2. If `attachment_paths` is non-empty, [`ingest_attachments()`](#chat-attachments) stages each one
   for `conversation.id` before anything else, so they are visible to this turn's own retrieval
   (step 4a) as well as later turns in the same conversation.
3. The `user` `Message` (the original `message`) is added and flushed immediately, so its `id` is
   available as the `source_id` for any `Fact` recorded from this turn (step 6).
4. `_truncate_history()` keeps the most recent whole messages whose combined `content` length
   fits within `chat_history_max_chars` (always keeping at least the latest message), as a list
   of `HistoryTurn(role, content)`. A cheap existence check on `ConversationAttachment` for
   `conversation.id` decides whether step 5a's attachment search runs at all, avoiding an extra
   `embed()` call for the common case of a conversation with no attachments.
5. Steps 5a and 5b have no data dependency on each other (both depend only on `history` and
   `message`), so they run concurrently via `asyncio.gather`:
   - **5a.** If there is any history, `build_query_rewrite_prompt(history, message)` is sent to
     `provider.generate(..., system=QUERY_REWRITE_SYSTEM_PROMPT)`, asking the model to rewrite the
     latest message into a standalone search query (resolving references like "this project"
     using the history). The result is used as the retrieval query. With no history, the original
     `message` is used directly and this call is skipped. Either way,
     `search_chunks(db, provider, query, settings.chat_retrieval_limit)` runs next; if this
     conversation has any attachments (step 4's check), `search_attachment_chunks(db, provider,
     conversation.id, query, settings.chat_attachment_retrieval_limit)` also runs and its results
     are appended after the corpus results.
   - **5b. Fact detection**: `build_fact_update_prompt(history, message)` is sent to
     `provider.generate(..., system=FACT_UPDATE_SYSTEM_PROMPT, format=FactUpdateResult.model_json_schema())`.
6. The parsed `FactUpdateResult` (from 5b) is passed to
   [`record_fact()`](#chat-memory-facts), which inserts a new `Fact` row with `status="pending"`
   if the message states a fact worth recording. This call happens on every turn, regardless of
   history. If a `Fact` was recorded, [`describe_fact()`](#fact-resolution) builds the
   `pending_fact` returned to the frontend (step 11) so it can be surfaced for confirmation. Note
   this fact is not yet `"confirmed"`, so it does not affect this same turn's answer (step 8); see
   the [improvements backlog](../contributing/improvements-backlog.mdx#a-newly-recorded-fact-does-not-affect-the-answer-in-the-same-turn).
7. `known_facts = build_known_facts_block(db, query)` (see [chat memory](#chat-memory-facts)), only
   considering `Fact` rows with `status == "confirmed"`.
8. `build_chat_prompt(message, sources, history, known_facts)` builds, in order: a "Conversation so
   far" block (if `history` is non-empty), the retrieved chunks numbered `[1]`, `[2]`, ... each
   prefixed with its document path and section (an attachment's is prefixed `"attached file: "`),
   the "Known facts" block (if any), and finally `Question: <message>`. The Known facts block is
   placed immediately before the question, closest to where the model generates its answer.
9. `provider.generate(prompt, system=CHAT_SYSTEM_PROMPT)`, **no** `format=`: the answer is free
   text, expected to cite context entries as `[n]`. `CHAT_SYSTEM_PROMPT` instructs the model that a
   Known fact overrides any numbered context entry giving a different value for the same subject,
   for any question about that subject, not only ones phrased as a correction.
10. Persists the `assistant` `Message` (the answer, with `sources` as
    `[{"document_path", "section", "project_name"}, ...]`, `is_attachment` not included in the
    snapshot) on the conversation, and commits.
11. Returns `conversation_id`, the answer text, the combined `RetrievedChunk` list as `sources`,
    `pending_fact` (from step 6, `None` if no fact was recorded this turn), and `attachments` (from
    step 2, `[]` if `attachment_paths` was empty).

## Chat memory (Facts)

`app/chat/memory.py`:

### `resolve_project_by_name(db, name)`

Case-insensitive exact match of `name` against `Project.name`. Returns the matching `Project`, or
`None` if `name` is blank or no project matches (chat never creates a `Project`).

### `record_fact(db, update, source_message_id)`

`update` is a `FactUpdateResult` (see [LLM integration](./llm-integration.mdx)). Returns `None`
(recording nothing) if `update.should_record` is `false`, or `subject`/`predicate` is blank, or
both `object` and `value` are blank after stripping. Otherwise:

1. `resolve_project_by_name(db, update.project)` gives an optional `project_id` used to scope
   resolution.
2. [`resolve_entity_ref(db, project_id, update.subject, {}, document_id=None)`](#project-resolution)
   resolves the subject, falling back to a `Topic` with `document_id=None` if nothing matches.
3. If `update.object` is non-blank, the object is resolved the same way, scoped to `project_id`,
   except: if `project_id` is `None` and the subject itself resolved to a `Project`
   (`subject_type == "project"`), the object is resolved scoped to that project
   (`subject_id`) instead. This covers a `FactUpdateResult` with an empty `project` field whose
   subject names the project directly, so the object resolves to the same entity as a prior
   update for the same fact that did have `project` set, rather than a separate, unscoped one. If
   `update.object` is blank, `object_type`/`object_id` are `None` and the row's `value` is
   `update.value`.
4. Before inserting, checks for an existing `Fact` row with the same `(subject_type, subject_id,
   predicate, object_type, object_id, value)` and `status` in `("pending", "confirmed")`. If one
   exists, returns `None` without creating a row, this is the case where `fact_update` re-asserts
   a fact that was already recorded (see [improvements
   backlog](../contributing/improvements-backlog.mdx#fact-updates-can-record-the-same-fact-more-than-once)).
   A different `value` or `object` for the same subject/predicate is not a duplicate and is still
   recorded as a new row (append-only history).
5. Otherwise, a new `Fact` row is created with `status="pending"` (`source_type="message"`,
   `source_id=source_message_id`) and flushed. Existing `Fact` rows for the same subject/predicate
   are never modified.

### `build_known_facts_block(db, query)`

Loads `Fact` rows with `status == "confirmed"`, ordered by `(subject_type, subject_id, predicate,
created_at)`, and groups consecutive rows by that key. Returns `None` if there are no confirmed
`Fact` rows.

For each group, resolves the subject's display name (`Person.name` / `Team.name` /
`Project.name` / `Topic.name`, depending on `subject_type`) and keeps the group only if that name,
or the `predicate`, appears as a case-insensitive substring of `query` — this is the relevance
filter that keeps the block from growing unbounded as the number of recorded facts grows. Returns
`None` if no group matches.

Each remaining group becomes one line: a group with a single row reads
`"<subject> <predicate>: <value>"`; a group with more than one row reads as a history, listing
each row's `value` (or resolved object name) with its `created_at` date, oldest to newest. The
lines are joined under a `"Known facts (current values; use these instead of any conflicting
numbered context entry):"` header, stating the override rule inline with the data.

### Fact resolution

A `Fact` recorded by [`record_fact()`](#record_factdb-update-source_message_id) starts as
`status="pending"` and is excluded from [`build_known_facts_block()`](#build_known_facts_blockdb-query)
until resolved, mirroring [project resolution](#project-resolution)'s pending/confirm pattern.

#### `describe_fact(db, fact)`

Returns `{"id": str(fact.id), "subject": <name>, "predicate": fact.predicate, "object": <name or
value>}`, resolving `subject`/`object` display names the same way as `build_known_facts_block()`
(`Person.name` / `Team.name` / `Project.name` / `Topic.name`). `object` is the resolved object's
name if `object_type`/`object_id` are set, otherwise `fact.value`. Used to build both the
`pending_fact` in the [`/chat`](#post-chat) response and the [`GET
/facts/pending`](#get-factspending) list.

#### `resolve_fact(db, fact, confirm)`

Sets `fact.status = "confirmed" if confirm else "rejected"` and returns the `Fact`. Called by
[`POST /facts/{id}/resolve`](#post-factsidresolve); the caller is responsible for committing.
